tensmos 2 microservicios, se prueban con docker ,desplegados en minikube,verificando con herarmmientas scs,El archivo de Kubernetes para order-service define dos cosas: cómo se ejecuta el servicio dentro del cluster, y cómo otros servicios pueden comunicarse con él.

Primero, el Deployment indica que Kubernetes debe crear un solo Pod que ejecute el contenedor order-service:CHANGEME. Ese contenedor ejecuta tu aplicación escuchando en el puerto 8001. Además, se configuran dos chequeos automáticos:

uno para saber cuándo la app está lista para recibir tráfico (readiness),

otro para saber si sigue viva o debe reiniciarse (liveness).

Luego, el archivo define un Service interno del tipo ClusterIP. Esto permite que cualquier otro servicio dentro del cluster pueda acceder a order-service usando el nombre order-service y el puerto 8001, sin exponerlo hacia el exterior.
En resumen: este YAML le dice a Kubernetes “corre order-service, mantenlo vivo, y hazlo accesible dentro del cluster”.


El archivo de Kubernetes para user-service tiene exactamente la misma estructura y propósito, solo que para el otro microservicio.

El Deployment crea también un Pod, basado en la imagen user-service:CHANGEME. Este contenedor corre la aplicación del usuario escuchando en el puerto 8000. Igual que en el otro servicio, se configuran sondas para que Kubernetes sepa si el servicio ya está listo para atender peticiones y si sigue funcionando correctamente.
Si algo se cuelga, Kubernetes lo reinicia automáticamente.

El Service asociado también es de tipo ClusterIP, lo que quiere decir que es accesible únicamente dentro del cluster. Otros Pods podrán comunicarse con este servicio usando el nombre user-service en el puerto **8000`.
En resumen: este YAML le indica a Kubernetes “mantén user-service ejecutándose y disponible para los demás servicios internos”.

Contenedor: es la imagen Docker, que contiene todo lo necesario para ejecutar tu aplicación (código, librerías, dependencias, etc.).

Ejemplo: order-service:v1, user-service:latest.

Esto se construye con un Dockerfile y un docker build.

Manifiesto de Kubernetes (.yaml): es un documento de configuración que le dice a Kubernetes cómo usar esos contenedores dentro del cluster.

Ejemplo: cuántas réplicas, qué puertos exponer, variables de entorno, probes de readiness/liveness, política de actualización, etc.

En otras palabras: el contenedor define el qué (la aplicación y su entorno), el manifiesto define el cómo (cómo correrlo dentro de K8s).
